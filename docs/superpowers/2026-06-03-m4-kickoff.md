## 2. Périmètre (referentiel.md §module 4)

**Objectif pédagogique** : utiliser les structures de données du JDK, comprendre les génériques, manipuler les données avec lambdas et streams.

### Chapitres (8) → `courses/docs/module-4-collections-generiques-lambdas/`
1. **`List` et `ArrayList`** — ajouts, suppressions, parcours, `LinkedList` vs `ArrayList`.
2. **`Set` et `HashSet`** — unicité, `equals` / `hashCode`, `TreeSet`.
3. **`Map` et `HashMap`** — clé/valeur, parcours, `getOrDefault`, `computeIfAbsent`.
4. **Itération moderne** — `Iterator`, `forEach`, vues `keySet` / `values` / `entrySet`.
5. **Génériques** — classes et méthodes génériques, bornes (`extends`, `super`), wildcards.
6. **`Comparable` et `Comparator`** — tri naturel vs custom, chaînage de comparators.
7. **Lambdas et interfaces fonctionnelles** — `Function`, `Predicate`, `Consumer`, `Supplier`, capture.
8. **Streams et `Optional`** — pipeline `map`/`filter`/`reduce`/`collect`, terminaux courants, `Optional` correctement utilisé.

### Exercices (12, en 3 sous-groupes) → `exercises/module-4-collections-generiques-lambdas/`
- **4.1 Collections** (4) — annuaire militaire (`Map`), set d'utilisateurs uniques, file de mission (`Deque`), groupement par grade.
- **4.2 Génériques et tri** (4) — `Stack<T>` custom, paire générique, tri d'une liste de `Soldat` (par grade puis nom), `Comparator` chaîné.
- **4.3 Lambdas et Streams** (4) — filtrage/projection d'une liste, agrégation (somme, moyenne, max), `groupingBy` / `partitioningBy`, recherche avec `Optional`.

## 4. 🔑 Nouveauté structurante du module 4 : les verrous M1–M3 sautent

Les modules 1 à 3 **interdisaient** les collections (tableaux uniquement), `equals`/`hashCode` écrits à la main, et `java.lang.Comparable<T>` générique (on avait fait une interface maison non générique en 3.4.3). **Le module 4 lève ces verrous** — c'est SON sujet :

- **Collections autorisées partout** : `List`/`Set`/`Map`/`Deque` remplacent les tableaux à capacité fixe. Les tests peuvent désormais s'appuyer dessus.
- **`equals`/`hashCode` enfin enseignés** (ch.2) : on peut **exiger de les écrire à la main** et tester le **contrat** (réflexivité, symétrie, cohérence avec `hashCode`, comportement dans un `HashSet`/`HashMap`). C'était explicitement repoussé depuis M3.
- **`Comparable<T>` générique + `Comparator`** (ch.6) : tri naturel et chaînage (`thenComparing`), là où 3.4.3 se limitait à une interface maison.
- **Programmation fonctionnelle** (ch.7-8) : lambdas, interfaces fonctionnelles, streams, `Optional`.

**Implication conception/test** : le code fonctionnel (streams, lambdas) **retourne des valeurs** → test direct AssertJ sur les résultats, simple. Mais deux points neufs à cadrer : (a) comment tester un **type générique custom** (`Stack<T>`) sans figer l'implémentation ? (b) faut-il **imposer** une solution en streams ou laisser le choix boucle/stream, sachant que le test n'observe que le résultat ?

## 5. Questions ouvertes à trancher au brainstorm (formateur)

- **Streams imposés ou libres ?** Le test n'observe que le résultat (une liste filtrée, une moyenne). Exige-t-on une solution en streams (et comment le vérifier sans réflexion fragile ?) ou laisse-t-on boucle/stream au choix, en valorisant l'idiomatisme via le critère formateur ?
- **`equals`/`hashCode` à la main** : jusqu'où teste-t-on le contrat ? (égalité simple ; symétrie ; cohérence `hashCode` ; mise en `HashSet`/clé de `HashMap`). Garde-fou : ne pas transformer ça en piège de coins juridiques.
- **Type générique custom** (`Stack<T>`, paire générique) : quel squelette dans le `starter/` (signatures génériques vides) ? Tests avec **plusieurs types** (`Stack<String>`, `Stack<Integer>`) pour prouver la généricité, sans dépendre des internals.
- **Fil rouge militaire** : `Soldat`/`Grade` (M3) reviennent en 4.2 (tri par grade puis nom) et 4.3 (agrégation, `groupingBy` par grade). Réutilise-t-on les classes M3 telles quelles (fournies dans le starter) ou en redéfinit-on une version locale au module ? (cohérence vs couplage inter-modules).
- **`Optional`** : comment tester un usage *correct* d'`Optional` (présence/absence) sans imposer une forme rigide ? Tests sur `isPresent`/valeur, pas sur la façon de l'obtenir.
- **Découpage des 8 chapitres** : 1 seule PR ou **2 PR de 4** (1-4 « collections & itération », 5-8 « génériques & fonctionnel ») — comme les 2×5 du module 3 ? 8 chapitres = gros volume.
- **`Comparator` vs interface maison de 3.4.3** : faut-il un chapitre/exo qui fait explicitement le pont (« voici le vrai `Comparable<T>` du JDK, voici ce que vous aviez fabriqué à la main ») pour ancrer la progression ?
