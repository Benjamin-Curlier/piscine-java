## 0. Nouveauté structurante de ce cycle

Les sous-groupes M1→M4 **interdisaient** la levée d'exceptions (cas limites gérés par sentinelle / `null` / liste vide). **5.1 lève ce verrou** : les exos **exigent** désormais de **lever** (`throw`) et les tests **vérifient le comportement d'exception**. Conséquence sur la rédaction des tests (vs M4) :

```java
import static org.assertj.core.api.Assertions.assertThatThrownBy;
import static org.assertj.core.api.Assertions.assertThatCode;
// ...
assertThatThrownBy(() -> ValidationSaisie.validerEntier("999", 0, 10))
    .isInstanceOf(IllegalArgumentException.class)
    .hasMessageContaining("10");
assertThatCode(() -> ValidationSaisie.validerEntier("5", 0, 10))
    .doesNotThrowAnyException();
```

On teste **le comportement observable** (type levé, message via `hasMessageContaining` — jamais `isEqualTo` sur un message JDK — et `getCause()` quand le chaînage est pertinent), **jamais l'implémentation interne** (pas de réflexion). Périmètre gradué public/privé : cf. spec chapitres §10.5.

## 2. Les 3 exercices

### 5.1.1 `saisie-defensive` — validateur générique — critère formateur `respect-consignes`

**Classe `ValidationSaisie`** (une seule méthode `static`, pas d'état) :

```
static int validerEntier(String texte, int min, int max)
//  1. texte == null  -> throw new NumberFormatException("...")   (garde EXPLICITE : sinon texte.trim() lèverait une NPE)
//  2. Integer.parseInt(texte.trim())   -> NumberFormatException si non numérique / overflow
//  3. valeur < min || valeur > max     -> IllegalArgumentException, message nommant min ET max + la valeur
//  4. sinon -> renvoie valeur
```

Décisions de cadrage : la méthode **`trim()`** la chaîne avant `parseInt` (robustesse) — d'où la **garde `null` explicite en étape 1** (`texte.trim()` sur `null` lèverait une `NullPointerException` ; on regroupe `null` avec les entrées illisibles → `NumberFormatException`). Sur entrée non numérique on **laisse remonter** la `NumberFormatException` de `parseInt` (on ne la rattrape pas — la consigne distingue les deux familles : illisible ≠ hors plage). Le message de l'`IllegalArgumentException` (hors plage) **contient les deux bornes** et la valeur fautive. Note : `NumberFormatException extends IllegalArgumentException`, donc les tests de format asserteront le **type précis** `NumberFormatException`, ceux de plage le type `IllegalArgumentException` levé en propre.

**Starter** : `return 0; // TODO` (compile, échoue).

**Tests publics** (≈8) : valeur au milieu de `[0,10]` → renvoyée ; égale à `min` → ok ; égale à `max` → ok ; `min-1` → `IllegalArgumentException` ; `max+1` → `IllegalArgumentException` ; `"abc"` → `NumberFormatException` ; message de l'IAE contient `"0"` et `"10"` ; cas valide → `assertThatCode(...).doesNotThrowAnyException()`.

**Tests privés** (≈8) : bornes exactes (`min-1`/`max+1` ko, `min`/`max` ok) ; plage d'une seule valeur (`min == max`) ; plage négative (`[-10,-1]`, `-5` ok) ; `trim` (`"  7  "` → `7`) ; signe explicite (`"+5"` → `5`, `"-3"` accepté si dans la plage) ; `null` → `NumberFormatException` ; overflow (`"99999999999"`) → `NumberFormatException` ; message contient la valeur fautive.

### 5.1.2 `effectif-invalide` — exception custom écrite par le stagiaire — critère formateur `demarche`

**Le stagiaire écrit `EffectifInvalideException extends RuntimeException`** (le cœur de l'exo, ch.5-4) avec **deux constructeurs** :

```java
public EffectifInvalideException(String message)                 { super(message); }
public EffectifInvalideException(String message, Throwable cause) { super(message, cause); }  // chaînage
```

**Classe `Unite`** — contrat décrit dans le `sujet.md` ; dans le **starter**, champs + constructeur + getters **fournis complets**, les deux `affecter` **à compléter** :

```
Unite(String nom, int capaciteMax)         // effectif initial = 0
String getNom()      int getCapaciteMax()   int getEffectif()

void affecter(int effectif)
//  effectif < 0 || effectif > capaciteMax  -> throw EffectifInvalideException(message nommant valeur + nom d'unité)
//  sinon -> met à jour effectif

void affecter(String effectifTexte)
//  Integer.parseInt échoue (NumberFormatException) -> throw new EffectifInvalideException("...illisible...", e)   // CHAÎNAGE
//  sinon -> délègue à affecter(int)  (la garde de plage s'applique, SANS cause)
```

L'overload `affecter(String)` **motive le chaînage** (`getCause()` = la `NumberFormatException` d'origine) — c'est le pont concret avec le ch.5-4 et le critère `demarche`.

**Starter (subtilité compile-vs-échoue)** : les tests publics référencent `EffectifInvalideException` ; pour que le starter **compile**, il **ship un stub minimal** `EffectifInvalideException` (constructeur `(String)` seul + `// TODO : ajouter le constructeur de chaînage (String, Throwable)`) et un `Unite` dont les `affecter` ne lèvent pas (corps `// TODO`). Résultat : **compile, tests publics rouges**. Le stagiaire écrit le constructeur de chaînage et la logique de levée. (Solution : `Unite.java` + `EffectifInvalideException.java` complets → 10 fichiers.)

**Tests publics** (≈8) : `affecter(0)` / milieu / `capaciteMax` → ok, `getEffectif` reflète ; `affecter(-1)` → `EffectifInvalideException` ; `affecter(capaciteMax+1)` → `EffectifInvalideException` ; type levé correct (`isInstanceOf(EffectifInvalideException.class)`) ; message contient la valeur fautive ; `affecter("12")` (texte valide) → `getEffectif() == 12` ; `affecter("abc")` → `EffectifInvalideException`.

**Tests privés** (≈8) : `EffectifInvalideException` **`isInstanceOf(RuntimeException.class)`** (unchecked, aucun `throws` requis) ; bornes exactes (`-1`/`capaciteMax+1` ko, `0`/`capaciteMax` ok) ; `affecter("abc")` → `getCause()` **instanceof `NumberFormatException`** (chaînage préservé) ; `affecter("-5")` (parseable mais hors plage) → `EffectifInvalideException` **sans cause** (`getCause() == null` — garde de plage, pas de parsing) ; message contient le **nom d'unité** ; `capaciteMax == 0` (seul `0` valide, `1` ko) ; **état inchangé après échec** (un `affecter` qui lève laisse `getEffectif` à la dernière valeur valide) ; séquence d'affectations valides → dernière valeur conservée.

### 5.1.3 `calculs-geometriques-exceptions` — refactor 2.3.3 avec gardes — critère formateur `idiomatisme`

**Classe `CalculsGeometriques`** — recopie locale `etnc.m5` du code de [`2.3.3`](../../../exercises/module-2-tableaux-chaines-methodes/2.3.3-refactor-calculs-geometriques/), **4 signatures imposées inchangées** (surcharge cercle/rectangle), `Math.PI` :

```
static double aire(double rayon)                  // garde rayon >= 0,  puis Math.PI * rayon * rayon
static double aire(double largeur, double hauteur) // garde largeur,hauteur >= 0, puis largeur * hauteur
static double perimetre(double rayon)             // garde rayon >= 0,  puis 2 * Math.PI * rayon
static double perimetre(double largeur, double hauteur) // garde >= 0, puis 2 * (largeur + hauteur)
```

Chaque méthode lève **`IllegalArgumentException`** si une dimension est **`< 0`** (le **zéro est autorisé** : aire/périmètre nuls, dégénérés mais valides). Le message **nomme le paramètre fautif** (`"rayon"` / `"largeur"` / `"hauteur"`). Critère `idiomatisme` : **un helper privé** `exigerPositif(double valeur, String nom)` mutualise la garde (pas de duplication), usage idiomatique de l'`IllegalArgumentException` du JDK (pas de type custom — recoupement évité avec 5.1.2 ; recopie `etnc.m5` cf. spec chapitres §10.6, choix `IllegalArgumentException` cf. §10.1).

**Starter** : 4 méthodes présentes (signatures, pour que la surcharge compile), corps `return 0; // TODO`.

**Tests publics** (≈8) : `aire(2.0)` ≈ `Math.PI*4` (`within(1e-9)`) ; `aire(3.0, 4.0) == 12.0` ; `perimetre(2.0)` ≈ `4*Math.PI` ; `perimetre(3.0, 4.0) == 14.0` ; `aire(-1.0)` → IAE ; `perimetre(-1.0)` → IAE ; `aire(-1.0, 4.0)` → IAE (largeur) ; `aire(3.0, -4.0)` → IAE (hauteur).

**Tests privés** (≈8) : `aire(0.0) == 0.0` et `perimetre(0.0) == 0.0` (zéro autorisé) ; rectangle à un côté nul `aire(0.0, 5.0) == 0.0` ; carré vu comme rectangle `aire(5.0, 5.0) == 25.0` ; `perimetre(-1.0, 4.0)` et `perimetre(3.0, -4.0)` → IAE ; message contient le nom du paramètre fautif ; type levé `isInstanceOf(IllegalArgumentException.class)` ; cercle rayon 1 (`aire`/`perimetre` corrects) ; `-0.0` traité comme `0` (autorisé — `-0.0 < 0` est `false` en Java).
