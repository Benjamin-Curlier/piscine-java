## 1. Décisions tranchées au brainstorm (2026-06-03)

1. **On teste des OBJETS, pas la sortie console.** Les tests **instancient** les classes (`new Point2D(...)`), **appellent les méthodes** et **assertent l'état / les valeurs de retour** avec AssertJ. Pas de `CaptureSortie`/`CaptureEntree` dans 3.1 (aucune E/S console).
2. **Squelette `starter/` à signatures publiques complètes à corps vide** (décision figée chapitres §10.1 pour 3.1/3.2) : classes, constructeurs et méthodes présents avec corps minimal (`return` par défaut / `// TODO`) + **Javadoc des contrats**. Les tests compilent dès le départ et restent stables. On évalue **la logique, pas le nommage**.
3. **Frontière 3.1 / 3.2 : « 3.1 douce, 3.2 stricte ».** En 3.1 : attributs **privés** + accesseurs `get`/`set` + méthodes de comportement, **sans validation d'invariants** (constructeurs et mutateurs acceptent tout). Toute la validation (rejet de valeurs, invariants gardiens, découvert interdit…) est **réservée au sous-groupe 3.2**. Évite redite et spoil.
4. **Nommage** (confirmé par les chapitres M3) : membres en **français**, accesseurs préfixés `get`/`set` (`getNom()`, `deposer()`, `getSolde()`, `aire()`…). Classes en `PascalCase`.
5. **`toString()` imposé et testé** sur les 3 exos (dans le `starter/`, vérifié en **test privé**), avec **`Locale.ROOT`** pour tout format à décimales (insensibilité à la locale de la JVM).
6. **`Point2D` en `double`** (x, y) : permet une vraie distance euclidienne (`Math.sqrt`) et l'override `toString` formaté.
7. **Difficulté / durée** : 3.1.1 `facile` (20) / 3.1.2 `facile` (25) / 3.1.3 `moyen` (30).

## 2. Spécification par exercice

Toutes les classes : attributs **privés**, accesseurs `get` pour les champs exposés, pas de `set` sauf besoin. Aucune validation d'entrée (cf. §1.3). Les **plancher/plafond** appliqués (ex. PV ≥ 0) sont du **calcul métier**, pas de la validation d'entrée, et restent donc autorisés en 3.1.

### 3.1.1 — point2d (`facile`, 20 min) — **gabarit du sous-groupe**
- **But pédagogique** : classe minimale, attributs privés, méthode mutatrice + méthode dérivée (calcul), override `toString`.
- **API publique imposée** (`Point2D`) :
  ```
  Point2D(double x, double y)
  double getX()
  double getY()
  void deplacer(double dx, double dy)   // translate : x += dx ; y += dy
  double distance(Point2D autre)         // distance euclidienne sqrt(dx²+dy²)
  String toString()                       // format "(1.0, 2.0)" via Locale.ROOT
  ```
- **Format `toString`** : `"(" + x + ", " + y + ")"` rendu en `Locale.ROOT` → ex. `(1.0, 2.0)`. Le contrat est donné dans la Javadoc du starter et illustré dans le sujet (sert de contrat de test).
- **Cas publics** : construction + getters ; `deplacer` puis getters ; `distance` cas nominal (ex. (0,0)→(3,4) = 5.0) ; `toString`.
- **Cas privés** : `distance` d'un point à lui-même = 0.0 ; coordonnées négatives ; `deplacer` cumulé (deux appels) ; `distance` symétrique (a.distance(b) == b.distance(a)) ; `toString` après `deplacer`.

### 3.1.2 — compte (`facile`, 25 min)
- **But pédagogique** : état mutable piloté par des méthodes ; **pas de garde de découvert** (réservée à 3.2 → `retirer` peut rendre le solde négatif).
- **API publique imposée** (`Compte`) :
  ```
  Compte(String titulaire, double soldeInitial)
  String getTitulaire()
  double getSolde()
  void deposer(double montant)           // solde += montant
  void retirer(double montant)           // solde -= montant (sans contrôle en 3.1)
  String toString()                       // "Compte de <titulaire> : <solde> €" (Locale.ROOT, 2 décimales)
  ```
- **Format `toString`** : `String.format(Locale.ROOT, "Compte de %s : %.2f €", titulaire, solde)`.
- **Cas publics** : construction + getters ; `deposer` ; `retirer` simple ; `toString`.
- **Cas privés** : enchaînement dépôt/retrait ; **retrait amenant un solde négatif** (autorisé ici — note pédagogique : la garde arrive en 3.2) ; solde initial à 0 ; `toString` avec décimales (`12.5` → `12.50`).

### 3.1.3 — soldat (`moyen`, 30 min)
- **But pédagogique** : plusieurs méthodes de comportement + **état booléen dérivé** ; touche militaire au service de la notion.
- **API publique imposée** (`Soldat`) :
  ```
  Soldat(String nom, String grade, int pointsDeVie)
  String getNom()
  String getGrade()
  int getPointsDeVie()
  void subirDegats(int degats)           // pointsDeVie -= degats, plancher à 0
  void soigner(int soin)                 // pointsDeVie += soin
  boolean estVivant()                    // pointsDeVie > 0
  String toString()                       // "<nom> (<grade>) — <pv> PV"
  ```
- **Cas publics** : construction + getters ; `subirDegats` cas nominal ; `soigner` ; `estVivant` vrai au départ ; `toString`.
- **Cas privés** : `subirDegats` au-delà des PV → **plancher 0** et `estVivant()==false` ; `soigner` après dégâts ; `subirDegats` puis `soigner` cumulés ; `estVivant` à PV exactement 0 (→ false) ; `toString` après dégâts.

## 3. Structure et constantes (les 3 exos)

Conforme à [`format-exercice.md`](../../format-exercice.md) §2. Chaque dossier `3.1.E-slug/` contient les 11 fichiers, **`tests-prives/` inclus pour les 3**.

- **Package** : `etnc.m3`. **`groupId`** : `etnc.piscine.m3`. **Dossier module** : `module-3-poo`. **Java 25**.
- **`starter/pom.xml`** : calqué sur module 2 (build-helper ajoute `../tests/src/test/java`). Dépendances JUnit Jupiter (5.11.0) + AssertJ (3.26.3). **Pas d'utilitaire `etnc.util`** (aucune capture de console).
- **`solution/pom.xml`** : identique + injecte **aussi** `../tests-prives/src/test/java`.
- **Classes** : `Point2D`, `Compte`, `Soldat`. Tests : `<Classe>Test` (public) / `<Classe>PriveTest` (privé).
- **Starter à signatures complètes** : chaque méthode présente, corps minimal compilable (`return 0;` / `return null;` / `// TODO`), **Javadoc du contrat** sur classe + méthodes. Les attributs privés sont déjà déclarés.

## 4. Énoncés (`sujet.md`)

Structure imposée par `format-exercice.md` §3 : Contexte (touche militaire) / Énoncé / Exemple / Contraintes / Ce qui sera vérifié / Pour aller plus loin. Vouvoiement, phrases courtes, pas d'humour (charte).

- **Contrainte commune à afficher** : « Complétez les méthodes du `starter/` **sans modifier les signatures publiques** : les tests s'appuient dessus. »
- **L'exemple** présente un petit scénario d'utilisation (`new …`, appels de méthodes, état attendu) qui sert de contrat — **sans livrer le corps des méthodes** (anti-spoil).
- **Anti-spoil** : ne pas écrire la formule complète de `distance` dans l'énoncé (la donner comme « distance euclidienne »), ne pas donner le code de `toString` (donner le format attendu).
- **Format `toString`** explicitement documenté dans l'énoncé (c'est un contrat de test).

## 6. Ordre d'implémentation

3.1.1 → 3.1.2 → 3.1.3 (difficulté croissante). Le premier exo (3.1.1 `point2d`) sert de **gabarit** : une fois son `pom.xml` (starter + solution), sa structure de tests et son `evaluation.yml`/`metadata.yml` validés en local, les suivants les réutilisent.

## 7. Hors périmètre (YAGNI)

- **Pas de validation d'invariants** ni d'exceptions (réservé à 3.2 ; les exceptions au M5). Les mutateurs acceptent tout.
- **Pas d'`equals`/`hashCode` écrits à la main** (M4). Les tests comparent l'état via les getters, pas par égalité d'objets.
- **Pas d'héritage / interface / record / enum** (sous-groupes 3.3 et 3.4).
- **Pas de collections génériques, lambdas, streams** (M4).
- **Pas d'E/S console** (ni `Scanner`, ni `System.out` testé) : on teste les objets directement.
