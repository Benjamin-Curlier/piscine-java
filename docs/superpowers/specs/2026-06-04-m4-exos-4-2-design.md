## 3. Les 4 exercices

### 4.2.1 `pile-generique` — gabarit — idiomatisme (desc : « généricité idiomatique : `<T>` propagé, pas d'`Object`/cast »)
**Classe `Pile<T>`** (champ `private final List<T> elements = new ArrayList<>();` — non observable).
```
void empiler(T element)   // elements.add(element)
T depiler()               // pile vide -> null ; sinon retire et renvoie le dernier (LIFO) : elements.remove(elements.size()-1)
T sommet()                // pile vide -> null ; sinon elements.get(elements.size()-1) (ne retire pas)
boolean estVide()         // elements.isEmpty()
int taille()              // elements.size()
```
**Starter** : en-tête générique COMPLET fourni (`public class Pile<T> { ... }`), corps `return null; / return true; / return 0; / // TODO` (compile mais échoue).
**Tests publics** (≈4) : `Pile<String>` empiler "a","b" → `depiler` renvoie "b" puis "a" (LIFO), avec `String x = pile.depiler();` (affectation **sans cast**) ; `Pile<Integer>` empiler 1,2 → `depiler` 2 puis 1, avec `Integer n = pile.depiler();` (**prouve la généricité par compilation**) ; `sommet` ne change pas la taille ; `estVide`/`taille` cohérents.
**Tests privés** (≈3) : `depiler` sur pile vide → `null` (pas d'exception) ; `sommet` sur pile vide → `null` ; empiler puis tout dépiler → `estVide` true ; ordre LIFO sur une séquence de 4-5 éléments.

### 4.2.2 `paire-generique` — idiomatisme (desc : « deux paramètres de type, immuabilité, méthode générique »)
**Classe `Paire<A, B>`** (`final`, champs `private final A premier; private final B second;`).
```
Paire(A premier, B second)
A premier()
B second()
Paire<B, A> inverser()              // new Paire<>(second, premier)
static <X, Y> Paire<X, Y> de(X x, Y y)   // new Paire<>(x, y) — fabrique générique
```
**Starter** : en-tête générique complet, corps TODO (constructeur vide, accesseurs `return null;`, `inverser`/`de` `return null;`).
**Tests publics** (≈4) : `Paire<String,Integer>` → `premier()`/`second()` bons types/valeurs avec `String s = p.premier(); Integer i = p.second();` ; `inverser()` → `Paire<Integer,String>` composants échangés ; `de("x", 1)` infère les types et renvoie une paire correcte.
**Tests privés** (≈3) : immuabilité (`inverser()` ne modifie pas l'original, renvoie un nouvel objet — l'original garde `premier()` inchangé) ; `Paire<String,String>` fonctionne ; `inverser()` deux fois redonne l'ordre initial (valeurs égales).

**Tests publics** (≈4) : une `List<Soldat>` mélangée → `Collections.sort` → `containsExactly` dans l'ordre attendu (grade croissant `SOLDAT`<…<`LIEUTENANT`, puis nom alpha à grade égal) ; `compareTo` d'un grade inférieur vs supérieur → négatif (`assertThat(soldatSoldat.compareTo(soldatLieutenant)).isNegative()`) ; cas inverse → positif ; à grade égal, départage par nom (signe selon nom).
**Tests privés** (≈3) : même grade ET même nom → `compareTo` zéro ; tri stable sur grade égal (ordre par nom) ; cohérence `a.compareTo(b)` et `b.compareTo(a)` de signes opposés.

### 4.2.4 `comparator-chaine` — idiomatisme (desc : « `Comparator.comparing`/`thenComparing`/`reversed`, ne pas muter la source »)
**Fourni** : `Grade.java` + `Soldat.java` (record **complet**, sans `Comparable`). **À compléter** : `TriSoldats`.
```
static List<Soldat> parNom(List<Soldat> soldats)            // Comparator.comparing(Soldat::nom)
static List<Soldat> parGradePuisNom(List<Soldat> soldats)   // comparing(Soldat::grade).thenComparing(Soldat::nom)
static List<Soldat> parGradeDecroissant(List<Soldat> soldats) // comparing(Soldat::grade).reversed()
```
Chaque méthode renvoie une **nouvelle** liste triée (`List<Soldat> copie = new ArrayList<>(soldats); copie.sort(comparator); return copie;`) — **ne mute pas** la source.
**Starter** : `TriSoldats` stubs `return null; // TODO` (Soldat fourni complet).
**Tests publics** (≈3) : `parNom` → `containsExactly` ordre alpha ; `parGradePuisNom` → grade puis nom ; `parGradeDecroissant` → ordre des grades inversé.
**Tests privés** (≈3) : le chaînage départage 2 soldats de même grade par le nom ; la liste **source n'est pas mutée** (vérifier que l'ordre de la liste d'entrée est inchangé après appel) ; `parGradeDecroissant` inverse bien l'ordre ; liste à 1 élément renvoyée inchangée.
