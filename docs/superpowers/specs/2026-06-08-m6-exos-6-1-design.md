# Spec — #22 Exos 6.1 « Tests avec JUnit 5 » (écriture de tests, grading par mutation)

> Design du 2026-06-08. Sous-groupe **6.1** du module 6. Premier sous-groupe « écriture de tests » : le livrable du stagiaire est **la suite de tests**, gradée par le `MutationChecker` (#57, mergé PR #50).
> Pré-requis : chapitres M6 (mergés PR #49), **#57 MutationChecker** (mergé). Cadre : [`format-exercice.md`](../../format-exercice.md) §11bis (layout écriture-de-tests), spec chapitres [`2026-06-08-m6-chapitres-design.md`](2026-06-08-m6-chapitres-design.md) §10.1/§10.4/§10.5, kickoff [`2026-06-08-mutation-checker-kickoff.md`](../2026-06-08-mutation-checker-kickoff.md).

## 1. Objectif et périmètre

Produire les **4 exercices** du sous-groupe 6.1 dans `exercises/module-6-tests-git/`, tous de type **« écriture de tests »** (package `etnc.m6`, groupId `etnc.piscine.m6`). Le stagiaire reçoit une **classe correcte fournie** et écrit la suite de tests JUnit 5 / AssertJ qui la valide ; la moulinette grade par **mutation** (les tests doivent passer sur le correct et tuer chaque mutant caché).

**Hors périmètre** : le sous-groupe 6.2 (Git, cycle suivant), la fiche Projet binôme #3.

## 2. 🔑 Décision de conception : grading uniforme par mutation **[à valider]**

La spec chapitres §10.3 esquissait 6.1.3 (calculatrice TDD) et 6.1.4 (refactor) comme « le stagiaire livre **tests + code**, double grading ». Or l'infrastructure #57 rend le `MutationChecker` et les checkers normaux **mutuellement exclusifs** (`appliesTo`) : un exo est *soit* « écriture de tests » (mutation), *soit* « écriture de code » (tests de référence). Le **double grading** (noter à la fois le code ET les tests du stagiaire) demanderait un mécanisme supplémentaire non construit en #57.

**Résolution retenue (à confirmer)** : **les 4 exos de 6.1 sont uniformément « écriture de tests »** (impl fournie correcte, grading mutation). La dimension TDD/refactor vit dans le **sujet** et le critère **formateur `demarche`**, pas dans le grading du code stagiaire. Avantages : cohérence, réutilise directement #57, pas de nouveau mécanisme. Le « refactor » de 6.1.4 devient « écrire la suite de tests qui *sécuriserait* un refactor » (filet), ce qui sert le même objectif pédagogique. *(Alternative = construire le double grading : plus lourd, reporté ; signalé.)*

## 3. Roster des 4 exercices

Toutes les classes-sous-test sont **fournies correctes** dans `starter/src/main/java/etnc/m6/` (le stagiaire ne les modifie pas) et **dupliquées** à l'identique dans `solution/src/main/java/`. Le stagiaire complète `starter/src/test/java/etnc/m6/<Xxx>Test.java` (squelette avec TODO). La suite **modèle** complète vit dans `solution/src/test/java/`.

| Exo | Classe fournie | Ce que le stagiaire teste | Mutants (3–5) | Critère formateur |
|---|---|---|---|---|
| **6.1.1 `tests-classe-existante`** | `Temperature` (`celsiusVersFahrenheit`, `estPositive`) | cas nominaux + limites (0 °C, −40 °C) | `operateur-inverse` (×9/5→×5/9), `offset-absent` (+32 oublié), `borne-zero` (`>` au lieu de `>=`) | `demarche` |
| **6.1.2 `tests-parametres`** | `ClassdeNote` (`mention(int)` : intervalles de note → mention) | **`@ParameterizedTest`+`@CsvSource`** sur plusieurs paliers ; sujet **impose** le paramétrage | mutants décalant une borne d'intervalle (`>=`→`>`), un palier mal mappé, retour constant | `respect-consignes` (usage du paramétrage) |
| **6.1.3 `calculatrice-tdd`** | `Calculatrice` (`ajouter/soustraire/multiplier/diviser`, `diviser` par 0 **lève** `ArithmeticException`) | 4 opérations + le cas division par zéro (`assertThatThrownBy`) | op inversée, `diviser` sans garde /0 (retourne au lieu de lever), résultat constant | `demarche` (TDD) |
| **6.1.4 `refactor-sous-tests`** | `Rectangle` (recopie locale `etnc.m6` de 3.2.1 : invariants largeur/hauteur > 0, `aire`, `perimetre`) | invariants (rejet/correction des dimensions ≤ 0) + calculs | mutant sans garde d'invariant, `aire`=somme au lieu de produit, `perimetre` ×1 au lieu de ×2 | `demarche` (filet avant refactor) |

**Antériorité** : 6.1 → chapitres 6-1→6-5 (pourquoi, JUnit bases/avancé, AssertJ, TDD). `@ParameterizedTest`/`@CsvSource` (6.1.2) ⇐ ch.6-3 ✓. `assertThatThrownBy` (6.1.3) ⇐ ch.6-4 + acquis M5 ✓. Aucune notion Git (sous-groupe 6.2) ✓.

## 4. Layout par exo (format-exercice §11bis)

```
6.1.X-<slug>/
├── metadata.yml            # module:6, sous_groupe:"6.1", mode: ecriture-tests
├── sujet.md                # « écris la suite de tests de la classe fournie » + consignes
├── correction.md           # la suite modèle expliquée + quels mutants chaque test tue
├── evaluation.yml          # voir §5
├── starter/
│   ├── pom.xml             # JUnit+AssertJ test scope ; test sous src/test/java (emplacement standard)
│   └── src/
│       ├── main/java/etnc/m6/Xxx.java       # FOURNI correct (ne pas modifier)
│       └── test/java/etnc/m6/XxxTest.java   # squelette : 1-2 @Test montrés + TODO
├── solution/
│   ├── pom.xml
│   └── src/
│       ├── main/java/etnc/m6/Xxx.java       # = starter/main
│       └── test/java/etnc/m6/XxxTest.java   # suite MODÈLE complète
└── mutants/
    ├── <id-1>/etnc/m6/Xxx.java
    └── …                                    # 3 à 5, un par règle/cas limite
```
*Note layout* : contrairement aux exos M1-M5 (test sous `tests/` + `build-helper`), ici le test du stagiaire est à l'emplacement **standard** `starter/src/test/java` (c'est SON livrable) → le starter pom n'a pas besoin de `build-helper` ; un `./mvnw -f starter/pom.xml test` local teste contre l'impl fournie.

## 5. `evaluation.yml` **[adaptation actée]**
Le runtime émet **un** `CheckResult` d'id `mutation` (binaire). Plutôt que les 2 critères `tests-valides`+`mutants-tues` de §10.4 (qui supposeraient 2 checkers), un mapping 1:1 honnête :
```yaml
total: 20
seuil_reussite: 12
criteres:
  - id: mutation        # tests passent sur le correct ET tuent tous les mutants
    poids: 16
    type: automatique
  - id: style           # Checkstyle sur le code de test du stagiaire (advisory)
    poids: 2
    type: automatique
  - id: <demarche|respect-consignes>
    poids: 2
    type: formateur
```
*(Un vrai split tests-valides/mutants-tués suppose le moteur de note fractionnaire — tâche future, cf. spec #57 §3.)*

## 6. Extension `scripts/valider-solutions.sh` (filet CI #11b)
Aujourd'hui le script fait `mvn test` sur chaque `solution/`. Pour un exo écriture-de-tests, il faut vérifier que la **suite modèle** (`solution/src/test/java`) : (a) **passe** sur `solution/src/main` (correct), (b) **tue** chaque `mutants/<id>/` (échoue quand on substitue le mutant). Détection : présence du dossier `mutants/`. Mécanique : compiler modèle vs correct → run (attendu vert) ; pour chaque mutant, compiler vs mutant → run (attendu rouge). **Réutiliser la logique du `MutationChecker`** plutôt que dupliquer en bash : option privilégiée = un petit point d'entrée headless de la moulinette qui prend `--exo <path> --mode valider-ecriture-tests`, sinon un bloc bash dédié. **À détailler au plan** ; ne pas bloquer la production des 4 exos (le `MutationChecker` est déjà testé, #57).

## 7. Vérification (par exo, avant PR)
- **Suite modèle passe sur le correct** : `mvn -f solution/pom.xml test` vert.
- **Suite modèle tue tous les mutants** : pour chaque mutant, substituer l'impl et relancer la modèle → rouge (vérifié à la main ou via le script §6).
- **Squelette starter compile** mais **échoue/incomplet** : `mvn -f starter/pom.xml test` ne « valide » pas encore (TODO non faits) — cohérent TDD (le squelette montre 1 test, en demande d'autres).
- **Le MutationChecker grade correctement** : la modèle → OK ; un squelette trivial → FAIL (mutants survivants). Vérifiable via les ITs #57 (déjà couverts) ou un run console.
- **Lint** : `git clean -fdX exercises/module-6-tests-git/ ; bash scripts/lint-exercices.sh` 0 erreur (vérifier que le lint accepte le layout écriture-de-tests — **point d'attention** : le lint exige peut-être `tests/` ; adapter le lint si besoin, comme partie du cycle).

## 8. Workflow
- Branche `feature/m6-exos-6-1` (créée). 1 PR pour les 4 exos (sous-groupe = 1 PR, convention M2-M5).
- Production exo par exo ; le critère subtil = **6.1.2** (paramétrage imposé) et **6.1.3** (calculatrice + exception) → soin sur les mutants. Inline.
- Gate : suites modèles vertes + mutants tués + lint 0 + (si fait) `valider-solutions` étendu. CI 4 jobs verts.
- Clôture : `backlog.md` (#22 volet 6.1 fait), puis cycle **6.2**.

## 9. Critères d'acceptation
- [ ] 4 exos `6.1.1`→`6.1.4` complets (layout §4, `mode: ecriture-tests`, `mutants/` 3–5).
- [ ] Chaque suite **modèle** passe sur le correct et tue **tous** ses mutants.
- [ ] Chaque squelette starter compile ; sujet/correction/metadata/evaluation conformes.
- [ ] Antériorité tenue (ch.6-1→6-5, aucune notion Git) ; anti-spoil vs guidés des chapitres (Temperature/Celsius, mention, calculatrice, Rectangle — distincts des domaines guidés estPair/Celsius-guidé/pgcd/couleurs).
- [ ] `evaluation.yml` mappé sur le checker `mutation` (§5).
- [ ] Lint 0 erreur (layout écriture-de-tests accepté) ; CI verte.
- [ ] Décision « grading uniforme par mutation » (§2) actée/validée.
