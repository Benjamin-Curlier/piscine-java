## 0. Nouveauté structurante de ce cycle : l'I/O fichier testée par `@TempDir`

5.1 a levé le verrou des exceptions. 5.2 ajoute **le premier contact fichier** des exos. Deux conséquences :

1. **Les méthodes des exos prennent un `Path` et propagent `throws IOException`** (exception **checked**, pont ch.5-5/5-2). Elles n'ouvrent jamais un chemin en dur : le `Path` est **fourni par l'appelant** (le test). Les flux (`Files.lines`) sont fermés via **`try-with-resources`** (ch.3).
2. **Tests par `@TempDir`** (JUnit Jupiter, **déjà disponible**, aucune nouvelle dépendance) — §10.3 :

```java
import org.junit.jupiter.api.io.TempDir;
import java.nio.charset.StandardCharsets;
import java.nio.file.Files;
import java.nio.file.Path;
// ...
@Test
@DisplayName("compte les lignes d'un fichier de 3 lignes")
void compte_trois_lignes(@TempDir Path tempDir) throws Exception {
    Path fichier = tempDir.resolve("donnees.txt");
    Files.writeString(fichier, "alpha\nbravo\ncharlie\n", StandardCharsets.UTF_8);
    assertThat(CompteurLignes.compterLignes(fichier)).isEqualTo(3);
}
```

Règles de test (§10.3) — **non négociables** :
- Chaque test écrit/lit ses fichiers **dans le `@TempDir`** (nettoyé automatiquement, aucun fichier hors sandbox).
- **UTF-8 explicite** (`StandardCharsets.UTF_8`) côté test ET côté solution.
- **Fins de ligne `\n` explicites** dans les fixtures et assertions (portabilité Windows/Linux — jamais `System.lineSeparator()`).
- On asserte le **contenu** (`Files.readString` / `readAllLines` → AssertJ `isEqualTo` / `containsExactly` / `contains`) ou la **valeur de retour**, **jamais l'API** employée pour le produire.
- Les méthodes de test déclarent `throws Exception` (ou `throws IOException`) — elles manipulent des `Path`.

## 2. Les 4 exercices

### 5.2.1 `compteur-lignes` — `Files.lines` + `try-with-resources` — critère formateur `idiomatisme`

**Classe `CompteurLignes`** (méthodes `static`, pas d'état) :
```
static long compterLignes(Path fichier) throws IOException
//   nombre total de lignes du fichier (lecture en flux : Files.lines + try-with-resources)
static long compterLignesNonVides(Path fichier) throws IOException
//   nombre de lignes NON blanches (ligne.isBlank() == false)
```
Idiomatisme : **`Files.lines(fichier, StandardCharsets.UTF_8)` dans un `try-with-resources`** (flux fermé), `.count()` / `.filter(l -> !l.isBlank()).count()`. **Starter** : `return 0; // TODO`.

**Publics** (≈8) : fichier 3 lignes → `compterLignes` 3 ; même fichier → `compterLignesNonVides` 3 ; fichier vide → 0 et 0 ; lignes vides intercalées (`"alpha\n\nbravo\n\n\ncharlie\n"`) → total 6, non-vides 3 ; ligne unique sans `\n` final → 1 ; lignes blanches (espaces/tabs) comptées par `compterLignes` mais pas par `compterLignesNonVides` ; accents UTF-8 préservés (lecture correcte).
**Privés** (≈8) : fichier uniquement blanc → non-vides 0 ; présence/absence de `\n` final donne le même total ; gros fichier (50 lignes) ; flux bien fermé (pas de fuite — implicite via try-with-resources) ; une seule ligne vide `"\n"` → total 1, non-vides 0 ; etc.

### 5.2.2 `copie-fichier` — `Files.copy` octet-exact — critère formateur `respect-consignes`

**Classe `CopieFichier`** :
```
static void copier(Path source, Path destination) throws IOException
//   copie le contenu de source vers destination, EN ÉCRASANT destination si elle existe,
//   en préservant le contenu octet pour octet (Files.copy + StandardCopyOption.REPLACE_EXISTING).
```
Consigne (critère `respect-consignes`) : **ne pas décoder/ré-encoder** (pas de `readAllLines`+`write` qui pourrait altérer la fin de ligne) — copie binaire via `Files.copy`. **Starter** : corps vide `// TODO`.

**Publics** (≈8) : copie → `Files.readString(destination)` == contenu source ; contenu accentué (UTF-8) préservé ; source **inchangée** après copie ; fichier vide → destination vide ; multi-lignes avec `\n` préservées ; écrasement d'une destination existante (REPLACE_EXISTING) → destination prend le contenu source.
**Privés** (≈8) : modifier la source après copie n'affecte pas la destination (indépendance) ; pas de `\n` ajouté en fin (octet-exact : source sans `\n` final → destination sans `\n` final) ; copie d'un fichier à une seule ligne ; recopie successive (idempotence du contenu).

### 5.2.3 `journal-evenements` — append-only (`StandardOpenOption.APPEND/CREATE`) — critère formateur `demarche`

**Classe `Journal`** :
```
static void ajouter(Path journal, String evenement) throws IOException
//   ajoute "evenement\n" EN FIN de fichier sans écraser ; crée le fichier au 1er appel.
//   Files.writeString(journal, evenement + "\n", UTF_8, StandardOpenOption.CREATE, StandardOpenOption.APPEND)
static List<String> lire(Path journal) throws IOException
//   renvoie toutes les lignes (Files.readAllLines, UTF_8) ; LISTE VIDE si le journal n'existe pas encore
//   (garder un Files.exists(journal) avant de lire).
```
Démarche (critère) : comprendre que l'écriture par défaut **écrase**, et qu'il faut **`CREATE` + `APPEND`** pour un journal ; gérer le cas « journal absent » sans exception. **Starter** : `ajouter` corps vide ; `lire` `return List.of(); // TODO`.

**Publics** (≈8) : `ajouter` une fois → `lire` renvoie `["evt"]` ; `ajouter` deux fois → `lire` renvoie `["e1","e2"]` dans l'ordre ; fichier créé au 1er `ajouter` (`Files.exists`) ; `lire` sur journal jamais créé → liste vide ; chaque `ajouter` ajoute exactement une ligne ; accents préservés ; le contenu existant est conservé (append, pas overwrite).
**Privés** (≈8) : 10 `ajouter` → 10 lignes ordonnées ; `ajouter` sur un fichier pré-rempli (créé via `Files.writeString`) ajoute **après** ; événement vide `""` → ajoute une ligne vide (total +1) ; `lire` ne *trim* pas (lignes brutes) ; alternance ajouter/lire cohérente.

### 5.2.4 `lecture-config` — parser `clé=valeur` à la main — critère formateur `idiomatisme`

**Classe `LectureConfig`** (⚠️ **sans `java.util.Properties`** — réservé au ch.7, cf. §10.4) :
```
static Map<String,String> lireConfig(Path config) throws IOException
//   lit le fichier (readAllLines, UTF_8) ; pour chaque ligne :
//     - ignorer les lignes blanches et celles commençant par '#'
//     - couper sur le PREMIER '=' (ex. ligne.indexOf('=') ou split("=", 2))
//     - trim sur la clé ET la valeur
//   renvoie la Map clé -> valeur (en cas de clé dupliquée : la DERNIÈRE gagne).
static String lireValeur(Path config, String cle, String valeurParDefaut) throws IOException
//   délègue à lireConfig : renvoie la valeur de la clé, ou valeurParDefaut si absente (Map.getOrDefault).
```
Idiomatisme (critère) : `split("=", 2)` / `indexOf` pour ne couper qu'au premier `=`, `Map.getOrDefault`, ignorer commentaires/blancs proprement. **Starter** : `lireConfig` `return Map.of(); // TODO` ; `lireValeur` `return null; // TODO`.

**Publics** (≈8) : config `"langue=fr\nniveau=3\n"` → `lireValeur(..,"langue",..)`=="fr", `"niveau"`=="3" ; clé absente → `valeurParDefaut` ; `lireConfig` renvoie la Map des 2 paires ; ligne de commentaire `"# titre"` ignorée ; ligne blanche ignorée ; espaces autour de clé/valeur *trimés* (`"  a = b  "` → `a`->`b`) ; valeur vide `"cle="` → `""`.
**Privés** (≈8) : `=` dans la valeur (`"url=http://x?a=b"` → valeur `"http://x?a=b"`, coupe au 1er `=`) ; clé dupliquée → dernière gagne ; ligne sans `=` ignorée ; clés sensibles à la casse ; `lireValeur` délègue bien à `lireConfig` ; config vide → Map vide + défaut renvoyé ; mélange commentaires/blancs/paires.
