# Spec — #22 Exos 6.2 « Git en pratique » (grading par état du repo)

> Design du 2026-06-08. Dernier sous-groupe d'exos du module 6 (et du parcours). 3 exercices.
> Pré-requis : chapitres M6 6-6→6-8 (mergés PR #49). Cadre : [`format-exercice.md`](../../format-exercice.md), spec chapitres §10.2 (grading Git par état du repo, validé formateur).

## 1. Objectif et périmètre

Produire les **3 exercices** du sous-groupe 6.2 dans `exercises/module-6-tests-git/`, sur le **workflow Git**. Un exo Git ne « retourne » rien : il **transforme l'état d'un dépôt**. On grade donc par **assertions sur l'état d'un repo-bac-à-sable** isolé.

**Hors périmètre** : la fiche Projet binôme #3, l'extension `valider-solutions.sh` (tâche dédiée).

## 2. 🔑 Forme du rendu : le stagiaire écrit du **code Java qui pilote git** **[à valider]**

Le nœud de conception. Un exo Git doit être **gradable automatiquement**, or la moulinette grade en exécutant des tests JUnit contre du code Java. Trois options envisagées :
- (A) **Le stagiaire écrit une méthode Java qui orchestre les opérations git** (via un helper `GitCommandes` fourni, thin wrapper `ProcessBuilder` sur `git`). Nos tests de référence montent un repo-jouet `@TempDir`, appellent sa méthode, et **assertent l'état résultant** (nombre de commits, branche, absence de marqueur de conflit, message, fichier présent). **Réutilise les checkers existants** (`PublicTestChecker`/`PrivateTestChecker`) — **zéro nouvelle infra moulinette.**
- (B) Le stagiaire écrit une **séquence de commandes git** (script) que le harnais rejoue. Ne rentre pas dans le modèle « code + tests » de la moulinette, et exécuter un script arbitraire pose un souci de sécurité/portabilité. Demanderait une nouvelle infra.
- (C) Exos **semi-guidés**, grading auto minimal, l'essentiel évalué par le formateur. Peu reproductible.

**Résolution retenue (à confirmer) : option A.** Avantages : automatique, déterministe, réutilise toute l'infra (comme les exos M1-M5), isolé du repo de rendu (`@TempDir`). **Compromis pédagogique assumé** : le stagiaire pratique git en **séquençant des opérations depuis Java** plutôt qu'en tapant les commandes au shell. Les *concepts* (staging, commit atomique, branche, merge, résolution de conflit, PR) sont exactement ceux des chapitres ; seul le *medium* diffère (Java au lieu du terminal). Le helper `GitCommandes` rend ça léger : le stagiaire écrit `git.run("add", "."); git.run("commit", "-m", "...")`, ce qui **mime fidèlement** les commandes du chapitre 6-6/6-7/6-8.

## 3. Le helper fourni `GitCommandes` (auto-contenu, dans chaque starter)

Classe **fournie** (`starter/src/main/java/etnc/m6/GitCommandes.java`, « ne pas modifier »), sans dépendance à la moulinette :
```java
public final class GitCommandes {
    private final Path repo;
    public GitCommandes(Path repo) { this.repo = repo; }
    /** Lance `git <args>` dans le repo et renvoie la sortie standard (lève si exit != 0… sauf merge en conflit). */
    public String run(String... args) { /* ProcessBuilder("git", args).directory(repo)… */ }
    public String log() { return run("log", "--oneline"); }
    public String brancheCourante() { return run("rev-parse", "--abbrev-ref", "HEAD"); }
}
```
- Configure `user.name`/`user.email` à l'init (déterminisme), encodage UTF-8, draine stderr (gotcha MinGit déjà connu de `ProcessGitClient`).
- Le **même helper** sert au stagiaire (pour agir) et aux tests de référence (pour préparer le repo-jouet + inspecter l'état).
- **git requis au runtime** : présent en CI (ubuntu), chez le formateur (git portable), dans le bundle standalone (MinGit). Les tests forkent git — lents mais déterministes (comme les `@Tag("git")` existants).

## 4. Roster des 3 exercices

| Exo | Le stagiaire écrit | Grading (état du repo) | Critère formateur |
|---|---|---|---|
| **6.2.1 `commit-propre`** | `creerHistorique(GitCommandes g)` : init (fait), ajoute 2 fichiers en **2 commits atomiques** avec messages à l'impératif | nb de commits == 2 ; chaque fichier dans le bon commit ; messages non vides/non « wip » | `respect-consignes` |
| **6.2.2 `resolution-conflit`** | `resoudreConflit(GitCommandes g)` : sur un repo **où un merge a déjà laissé un conflit** sur `README.md`, supprimer les marqueurs, garder le contenu fusionné, `add`+`commit` | aucun marqueur `<<<<<<<`/`=======`/`>>>>>>>` résiduel ; merge conclu (1 commit de merge) ; contenu attendu | `demarche` |
| **6.2.3 `pull-request-fictive`** | `preparerPullRequest(GitCommandes g)` : créer une branche `feature/...`, y commiter, revenir sur `main`, **merger** la branche, et écrire un `PULL_REQUEST.md` (titre + description + checklist) | branche créée + mergée dans `main` (sans marqueur) ; `PULL_REQUEST.md` présent et structuré (sections attendues) | `respect-consignes` |

**Antériorité** : 6.2 → chapitres 6-6 (add/commit/log), 6-7 (branches/merge/conflit), 6-8 (PR). Aucune notion de test JUnit non vue. ✓

## 5. Layout par exo (format-exercice classique, modèle « écriture de code »)

```
6.2.X-<slug>/
├── metadata.yml            # module:6, sous_groupe:"6.2"
├── sujet.md · correction.md · evaluation.yml
├── starter/
│   ├── pom.xml
│   └── src/main/java/etnc/m6/
│       ├── GitCommandes.java     # FOURNI (helper git auto-contenu)
│       └── <Tache>.java          # méthode à compléter par le stagiaire (corps TODO)
├── tests/                  # tests publics (montent un repo-jouet @TempDir, appellent la méthode, assertent l'état)
│   └── src/test/java/etnc/m6/<Tache>Test.java
├── tests-prives/           # cas limites (message exact, branche, pas de marqueur résiduel)
│   └── src/test/java/etnc/m6/<Tache>PriveTest.java
└── solution/
    ├── pom.xml
    └── src/main/java/etnc/m6/{GitCommandes,<Tache>}.java   # solution de référence
```
- **Pas de `mutants/`** : ce sont des exos « écriture de code » classiques → checkers normaux (`compile`/`tests-publics`/`tests-prives`/`style`), le `MutationChecker` ne s'active pas (`appliesTo` false sans `mutants/`). Aucune infra nouvelle.
- `evaluation.yml` standard : tests-publics 8 + tests-prives 8 + style 2 + 1 formateur 2 = 20.
- Le `pom.xml` starter ajoute `tests/` via `build-helper` (comme M1-M5).
