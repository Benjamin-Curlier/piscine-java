## 1. Décisions tranchées au brainstorm (2026-06-03)

1. **On teste des OBJETS** (instanciation + assertions AssertJ sur état/retours), pas la sortie console. Pas de `CaptureSortie`/`CaptureEntree`.
2. **Squelette `starter/` à signatures publiques complètes à corps vide** (décision figée §10.1 pour 3.1/3.2) : classes, constructeurs, méthodes présents, corps minimal compilable + **Javadoc des contrats** (y compris la règle de validation). On évalue la logique, pas le nommage.
3. **Convention de rejet SANS exception** (les exceptions sont au module 5) — conforme au chapitre 3-2 (« une valeur refusée laisse l'objet inchangé : on corrige ou on ignore ») :
   - **Setter** recevant une valeur invalide → **ignore** la demande, l'objet reste **inchangé** (pas de message console : on teste l'état, pas la sortie).
   - **Constructeur** recevant une valeur invalide → **corrige** vers une valeur valide **documentée** (il n'y a pas d'état antérieur à préserver). La valeur de correction fait partie du contrat (Javadoc + sujet).
   - **Méthode métier** (`retirer`) → **refuse** l'opération invalide, objet inchangé.
   Les tests assertent l'état **corrigé** (constructeur) ou **inchangé** (setter/méthode).
4. **Surcharge de constructeurs + chaînage `this(...)`** (chapitre 3-3) exercés et **testés** dans `Rectangle` (constructeur carré `Rectangle(double cote)` qui chaîne `this(cote, cote)`).
5. **3.2.2 prolonge le `Compte` de 3.1** : même API, mais **avec gardes**. C'est le fil pédagogique « 3.1 douce → 3.2 stricte ». La classe se nomme `CompteProtege` (slug `compte-protege`) pour ne pas se confondre avec l'exo 3.1.2.
6. **Nommage** : membres en **français**, accesseurs `get`/`set` ; classes `PascalCase`. `toString` imposé et testé, `Locale.ROOT` pour tout format à décimales.
7. **Difficulté / durée** : 3.2.1 `facile` (30) / 3.2.2 `moyen` (30) / 3.2.3 `moyen` (30).

## 2. Spécification par exercice

Toutes les classes : attributs **privés**. Validation selon §1.3. Aucune exception levée. `prerequis: []`.

### 3.2.1 — rectangle (`facile`, 30 min) — **gabarit du sous-groupe**
- **But pédagogique** : invariants (`largeur > 0`, `hauteur > 0`), validation au constructeur (correction) et au setter (refus), **surcharge de constructeurs** + **chaînage `this(...)`**.
- **API publique imposée** (`Rectangle`) :
  ```
  Rectangle(double largeur, double hauteur)   // toute dimension <= 0 est corrigée à 1.0
  Rectangle(double cote)                        // carré ; chaîne via this(cote, cote)
  double getLargeur()
  double getHauteur()
  void setLargeur(double largeur)               // <= 0 : ignoré (objet inchangé)
  void setHauteur(double hauteur)               // <= 0 : ignoré (objet inchangé)
  double aire()                                  // largeur * hauteur
  double perimetre()                             // 2 * (largeur + hauteur)
  String toString()                              // "Rectangle 3.0 x 4.0" (Locale.ROOT)
  ```
- **Contrat de correction** : toute dimension `<= 0` passée à un constructeur devient `1.0`. Le constructeur carré passe par `this(cote, cote)` (donc bénéficie de la même correction).
- **Format `toString`** : `String.format(Locale.ROOT, "Rectangle %s x %s", largeur, hauteur)` → ex. `Rectangle 3.0 x 4.0`.
- **Cas publics** : construction nominale + getters ; `aire` ; `perimetre` ; constructeur carré (`getLargeur() == getHauteur()`) ; `setLargeur` valide ; `toString`.
- **Cas privés** : constructeur avec `largeur <= 0` → `1.0` ; constructeur avec `hauteur <= 0` → `1.0` ; carré avec `cote <= 0` → `1.0 x 1.0` ; `setLargeur(0)` ou négatif → **inchangé** ; `aire` recalculée après un `setHauteur` valide.

### 3.2.2 — compte-protege (`moyen`, 30 min) — **prolonge 3.1.2**
- **But pédagogique** : ajouter des **gardes** à un objet à état mutable ; garde de découvert ; refus laissant l'objet inchangé.
- **API publique imposée** (`CompteProtege`) :
  ```
  CompteProtege(String titulaire, double soldeInitial)   // soldeInitial < 0 corrigé à 0.0
  String getTitulaire()
  double getSolde()
  void deposer(double montant)    // montant <= 0 : ignoré (inchangé)
  void retirer(double montant)    // montant <= 0 OU montant > solde : refusé (inchangé)
  String toString()                // "Compte de <titulaire> : <solde> €" (Locale.ROOT, 2 décimales)
  ```
- **Contrat** : `soldeInitial` strictement négatif est corrigé à `0.0`. `deposer` ignore un montant `<= 0`. `retirer` refuse un montant `<= 0` **ou** supérieur au solde (jamais de solde négatif).
- **Format `toString`** : identique à 3.1.2 — `String.format(Locale.ROOT, "Compte de %s : %.2f €", titulaire, solde)`.
- **Cas publics** : construction + getters ; `deposer` nominal ; `retirer` nominal ; `toString`.
- **Cas privés** : `soldeInitial` négatif → `0.0` ; `deposer` d'un montant `<= 0` → inchangé ; `retirer` supérieur au solde → **refusé** (inchangé) ; `retirer` égal au solde → solde `0.0` (autorisé) ; `retirer` d'un montant `<= 0` → inchangé.

### 3.2.3 — date (`moyen`, 30 min)
- **But pédagogique** : bornage d'attributs (invariants de plage) au constructeur et au setter ; formatage `toString` avec zéro-padding.
- **API publique imposée** (`Date`) :
  ```
  Date(int jour, int mois, int annee)   // jour borné à [1..31], mois borné à [1..12]
  int getJour()
  int getMois()
  int getAnnee()
  void setJour(int jour)   // hors [1..31] : ignoré (inchangé)
  void setMois(int mois)   // hors [1..12] : ignoré (inchangé)
  String toString()         // "07/03/2026" (zéro-padding)
  ```
- **Contrat de correction** : au constructeur, `jour < 1` → `1`, `jour > 31` → `31` ; `mois < 1` → `1`, `mois > 12` → `12`. L'année n'est pas contrainte. Date « simple » : pas de nombre de jours par mois ni d'année bissextile (réservé au « pour aller plus loin »).
- **Format `toString`** : `String.format("%02d/%02d/%04d", jour, mois, annee)` (insensible à la locale — entiers).
- **Cas publics** : construction nominale + getters ; `toString` (`new Date(7, 3, 2026)` → `"07/03/2026"`) ; `setJour` valide.
- **Cas privés** : `jour = 0` → `1` ; `jour = 40` → `31` ; `mois = 13` → `12` ; `setMois(0)` → inchangé ; `toString` zéro-padding sur jour et mois à un chiffre.

## 4. Énoncés (`sujet.md`)

Structure imposée par `format-exercice.md` §3 : Contexte (touche militaire) / Énoncé / Exemple / Contraintes / Ce qui sera vérifié / Pour aller plus loin. Vouvoiement, phrases courtes, pas d'humour (charte).

- **Contrainte commune** : « Complétez les méthodes du `starter/` **sans modifier les signatures publiques** : les tests s'appuient dessus. »

## 7. Hors périmètre (YAGNI)

- **Pas d'exceptions** (`throw`, `IllegalArgumentException`…) : réservé au module 5. La validation corrige/ignore/refuse.
- **Pas de messages console** dans les méthodes : on teste l'état, pas la sortie (le chapitre les évoque mais ici on s'en passe pour des tests robustes).
- **Date** : pas de nombre de jours par mois, pas d'année bissextile, pas de comparaison de dates (« pour aller plus loin »).
- **Pas d'`equals`/`hashCode` manuel** (M4), **pas d'héritage/interface/record/enum** (3.3/3.4), **pas de collections/streams** (M4).
