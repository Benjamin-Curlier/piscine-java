## 4. 🔑 Difficulté structurante du module 6 : la **méta-circularité**

Le module 6 **enseigne** JUnit, AssertJ et Git — or **la moulinette elle-même valide déjà via JUnit et un push git**. On teste le testeur, on grade un git workflow avec un outil git. C'est LA difficulté de conception du module ; deux fronts :

### 4.a — Exos 6.1 : le stagiaire **écrit des tests** (inversion du contrat habituel)
Dans tous les modules précédents, le stagiaire écrit du **code** validé par **nos** tests. En 6.1 (écrire des tests sur une classe existante, paramétrer, TDD), c'est l'inverse : **le livrable est le test**. Comment grader un test ?
- **Piste recommandée — la « mutation »** : on fournit la classe sous test, et la moulinette exécute les tests du stagiaire contre **deux** implémentations cachées : une **correcte** (les tests doivent **passer**) et une ou plusieurs **buggées/mutées** (les tests doivent **échouer** — preuve qu'ils détectent vraiment le bug). Un test qui passe sur tout, y compris le code buggé, ne « capture » rien → KO. C'est une mini-détection de mutation faite main, sans lib (pas de PIT). Les « tests privés » deviennent : *« les tests du stagiaire tuent-ils les mutants ? »*.
- **Pour le TDD-calculatrice (6.1.3)** : le stagiaire écrit tests **et** code. On grade le code par nos tests (contrat habituel) **et** on vérifie que ses tests existent et tuent des mutants. Double exigence assumée.
- **À trancher** : combien de mutants ? Mécanisme d'injection (classpath alterné côté moulinette, comme `PrivateTestChecker` qui swappe déjà des sources ? cf. `framework`/`runner`). Faut-il un nouveau `Checker` (`MutationChecker` / `StudentTestChecker`) ou est-ce exprimable avec l'existant ?

### 4.b — Exos 6.2 : grader un **workflow Git** sans sortie console testable
Un exo Git ne « retourne » rien — il **transforme l'état d'un dépôt**. Comment grader « le stagiaire a fait un commit propre / résolu un conflit / ouvert une PR » ?
- **Piste recommandée — assertions sur l'état du repo** : le test (que **nous** écrivons) ouvre le repo `@TempDir` du stagiaire et asserte : nombre de commits, existence d'une branche, merge sans marqueur de conflit (`<<<<<<<`), arbre attendu, message de commit conforme. On peut **réutiliser `console.git.GitClient` / `ProcessGitClient`** déjà écrits pour le MVP (lecture de branches, refs, log) plutôt que de réinventer.
- **« PR fictive » (6.2.3)** : pas de GitHub en local/standalone. Options : (a) simuler la PR comme un **merge de branche** local + un fichier `PULL_REQUEST.md` à rédiger (gradé sur sa présence/structure) ; (b) exercice **semi-guidé** dont la part automatique est légère et la part `formateur` plus lourde. À trancher — c'est l'exo le moins automatisable.
- **Risque outil** : la moulinette **utilise git** pour la séquence de rendu (`rendu/<sous-groupe>`). Un exo qui demande au stagiaire de manipuler des branches dans **son** workspace ne doit pas entrer en collision avec le repo de rendu de la piscine. Concevoir les exos Git pour qu'ils opèrent sur un **repo-bac-à-sable dédié** (créé par le starter dans un sous-dossier), pas sur le repo de la piscine.

**Implication** : 6.1 est techniquement le sous-groupe le plus subtil (mécanisme de mutation) → **Opus** en ultracode. 6.2 demande un arbitrage de conception fort (jusqu'où automatiser le Git) → brainstorm formateur **avant** la spec exos.

## 5. Questions ouvertes à trancher au brainstorm (formateur)

- **Mécanisme de grading des tests stagiaire (6.1)** : valider la piste « mutation maison » (§4.a) ? Combien de mutants par exo ? Nouveau `Checker` ou réutilisation de l'existant `framework`/`runner` ? *(Décision la plus structurante du module.)*
- **Niveau d'automatisation du Git (6.2)** : jusqu'où grader automatiquement vs part `formateur` ? La « PR fictive » : merge local + `PULL_REQUEST.md`, ou exo semi-guidé ? Confirmer le **repo-bac-à-sable** isolé du repo de rendu.
- **Refactor d'un exo M3 sous tests (6.1.4)** : lequel ? Un exo OO du module 3 (ex. `3.1.2-compte`, `3.2.1-rectangle`, `3.3.x`) dont on demande d'**écrire la suite de tests** (et éventuellement un petit refactor sûr sous filet de tests). **Recopie locale `etnc.m6`**, pas de dépendance inter-modules (fil rouge éprouvé M4/M5).
- **Calculatrice TDD (6.1.3)** : périmètre des opérations (4 opés + division par zéro qui **lève** — l'acquis exceptions M5 est dispo) ; impose-t-on l'ordre Red→Green→Refactor (gradé sur la démarche `formateur`) ?
- **`@ParameterizedTest` (6.1.2)** : sources (`@ValueSource`, `@CsvSource`, `@MethodSource`) — lesquelles enseigner/exiger ? `@CsvSource` est le plus lisible pour débutants.
- **Découpage des 8 chapitres en PR** : 1 seule PR (comme les 8 du M4) ou 2 PR (6-1→6-5 Tests/TDD / 6-6→6-8 Git) ? Le découpage 2 PR colle aux 2 sous-groupes d'exos.
- **Chapitres Git et extraits non-compilables** : les chapitres 6-6/6-7/6-8 contiennent des **commandes shell**, pas du Java. Le gate « compilation des extraits au JDK 25 » ne s'applique qu'aux extraits Java (ch.1-5). Adapter la revue adversariale en conséquence (vérifier la justesse des commandes git à la place).
- **Fil rouge** : exos indépendants (pas de fil rouge), comme M2-M5. Domaine militaire ponctuel quand il aide.
