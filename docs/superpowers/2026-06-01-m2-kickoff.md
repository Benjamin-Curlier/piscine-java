## 2. Périmètre (referentiel.md §4 module 2)

**Objectif pédagogique** : structurer du code en méthodes réutilisables ; manipuler tableaux 1D/2D et chaînes ; aborder la récursivité.

### Chapitres (5) → `courses/docs/module-2-tableaux-chaines-methodes/`
1. **Tableaux 1D** — déclaration, init, parcours, `length`, `Arrays.toString`.
2. **Tableaux 2D** — matrices, parcours imbriqué, applications (grilles, images N&B).
3. **Chaînes de caractères** — `String` immuable, méthodes courantes, `StringBuilder`.
4. **Méthodes** — signature, paramètres, retour, surcharge, portée, `static`.
5. **Récursivité** — cas de base, cas récursif, pile d'exécution, quand l'utiliser.

### Exercices (12, en 4 sous-groupes) → `exercises/module-2-tableaux-chaines-methodes/`
- **2.1 Tableaux** (4) : min/max/moyenne, inversion, recherche linéaire, rotation de matrice.
- **2.2 Chaînes** (3) : palindrome, comptage d'occurrences, formatage « ASCII art ».
- **2.3 Méthodes** (3) : bibliothèque de méthodes mathématiques, surcharge, **refactor d'un exercice du module 1 en méthodes**.
- **2.4 Récursivité** (2) : factorielle & puissance, parcours de matrice récursif.

## 4. Cadence recommandée (d'après rétro #17)

1. **Chapitres d'abord** (1 PR, branche `feature/m2-chapitres`) — les notions vues bornent ce que les exercices peuvent demander. Modèle : cycle #15.
2. **Exercices ensuite, 1 sous-groupe = 1 PR** (`feature/m2-exos-2-1`, `-2-2`, …), chacun branché depuis `main` après merge du précédent. Modèle : cycle #16.
3. Chaque PR : CI vert (4 jobs) **+ validation locale** (§1) avant push. MAJ du `docs/backlog.md` (#18) à chaque sous-groupe.

## 5. Questions ouvertes à trancher au brainstorm (formateur)

- **Antériorité « méthodes »** : garder « tout dans `main` » pour 2.1/2.2 (cohérent module 1) **ou** assumer que le chapitre 4 vient avant les exercices 2.3 et autoriser les méthodes seulement à partir de 2.3 ? (Impacte la façon de tester : tableaux/chaînes testés via `main` + E/S, ou via méthodes `static` appelées directement par les tests.)
- **Stratégie de test des tableaux/méthodes** : tester la **sortie console** (comme module 1) ou tester des **méthodes `static` retournant une valeur** (plus simple à asserter avec AssertJ, mais suppose une signature imposée dans le starter) ? Choix structurant pour les 12 exos.
