## 2. Périmètre (referentiel.md §module 5)

**Objectif pédagogique** : gérer correctement les erreurs, lire/écrire des fichiers, manipuler quelques formats texte courants.

### Chapitres (7) → `courses/docs/module-5-exceptions-io/` (`_category_.json` existe déjà, position 6, `collapsed:true` à passer `false`)
1. **Hiérarchie des exceptions** — `Throwable`, `Error`, `Exception`, `RuntimeException`, checked vs unchecked.

3. **`try-with-resources`** — interface `AutoCloseable`, fermeture déterministe.

## 4. 🔑 Nouveauté structurante du module 5 : le verrou « pas d'exceptions » saute

Les modules 1 à 4 **interdisaient** la levée d'exceptions (cas limites gérés par valeur sentinelle / `null` / `Optional`). **Le module 5 lève ce verrou** — c'est SON sujet :

- **Les exceptions sont enfin enseignées et exigibles** : on peut demander de **lever** (`throw`), **propager** (`throws`), **rattraper** (`try/catch`) et **tester** une exception (custom comprise). Les exos M1-M4 qui renvoyaient une sentinelle peuvent être **refactorés** pour lever une exception (cf. 5.1.3).
- **L'I/O fichier est neuf** : lecture/écriture, ressources fermées proprement (`try-with-resources`), chemins (`Path`/`Files`).

**Implication conception/test** : (a) tester une exception = `assertThatThrownBy` (comportement, pas réflexion) ; (b) tester l'I/O = écrire/lire dans un `@TempDir` puis asserter le contenu — déterministe, isolé ; (c) attention à l'antériorité : `try-with-resources` (ch.3) avant les exos I/O (5.2).

## 5. Questions ouvertes à trancher au brainstorm (formateur)

- **Checked vs unchecked pour débutants** : jusqu'où insister sur la distinction ? Les exceptions custom héritent-elles de `Exception` (checked, force le `throws`) ou `RuntimeException` (unchecked, plus souple) ? Recommandation à challenger : `EffectifInvalideException extends RuntimeException` (5.1.2) pour ne pas noyer le débutant sous les `throws`, mais enseigner les deux au chapitre 1.
- **Refactor d'un exo M2 (5.1.3)** : lequel ? Un exo de **validation/saisie** du M2 qui renvoyait une sentinelle → lever une exception sur entrée invalide. Candidats : un calcul géométrique avec garde, une conversion. À choisir au brainstorm (ne pas réimporter le code M2 — **version locale `etnc.m5`**, comme le fil rouge M4).
- **I/O classique (ch.5) vs NIO.2 (ch.6) dans les exos** : imposer NIO.2 (`Files.readAllLines`/`writeString`, plus simple et moderne) pour les exos 5.2, en montrant l'I/O classique surtout en cours ? Ou panacher ? Tester par le **contenu** du fichier (`@TempDir`), indépendamment de l'API choisie.
- **CSV/`.properties` (5.3)** : parsing **à la main** (`String.split`, `Properties`) sans lib tierce (cohérent offline/souveraineté) — confirmer. **JSON** : le référentiel dit « intro via une lib sans détailler » → **pas d'exo JSON** (mention en cours seulement), les exos restent CSV + `.properties` ? À trancher.
- **`@TempDir` et déterminisme** : confirmer le pattern (chaque test écrit ses fichiers dans le `@TempDir`, encodage UTF-8 fixe, fin de ligne `\n` explicite pour la portabilité Windows/Linux).
- **Découpage des 7 chapitres en PR** : 1 seule PR (comme les 8 du M4) ou 2 PR (1-4 exceptions / 5-7 I/O & formats) ? Préférence de cadence formateur.
- **Fil rouge** : exos indépendants (pas de fil rouge), comme M2/M3/M4. Domaine militaire ponctuel quand il aide.
